# Cross-Context Cache Invalidation

## Problem

The frontend caching system uses **context-isolated cache keys**. A merchant (TENANT context) PUT request and a public (PUBLIC context) GET request for the same logical data produce **different cache keys** and are stored in **different cache partitions**.

The auto-invalidation in `fetchWithCache` (`FlexibleApiSingleton.ts:1591-1604`) only clears the **current request's** cache key on non-GET requests. It does **not** cross-invalidate other contexts' caches that may serve the same data through a different endpoint.

This means: when a merchant updates bot config via `PUT /api/tenants/:tenantId/bot/config` (TENANT context), the public widget config at `GET /api/public/bot/config?tenantId=...` (PUBLIC context) remains stale for up to its TTL (5-15 minutes).

---

## Cache Key Format

All cache keys are generated by `generateCacheKey()` in `EnhancedFlexibleApiSingleton.ts:263-278`:

```
{baseKey}:{context}:{isolation}[:{tenantId}][:{userId}]
```

- `baseKey` = the `cacheKey` arg passed to `makePublicRequest` / `makeDefaultRequest`, or the URL if no cacheKey was provided
- `context` = `AppContext` enum value (e.g. `public`, `tenant`, `admin`, `user`, `customer`)
- `isolation` = `CacheIsolation` enum value (e.g. `public`, `tenant`, `admin`, `user`, `customer`)

### Concrete example from verbose logging

```
[PublicBotService] finalCacheKey: /api/public/bot/config?tenantId=tid-m8ijkrnk:public:public
[UnifiedCapabilityService] finalCacheKey: /api/tenants/tid-m8ijkrnk/effective-capabilities:public:public
```

When `cacheKey` is explicitly passed (e.g. `bot-widget-config-${tenantId}`), that becomes the baseKey instead of the URL.

## Context/Isolation by Domain Base

| Domain base | `defaultContext` | `defaultIsolation` | Typical endpoints |
|---|---|---|---|
| `PublicApiSingleton` | `public` | `public` | `/api/public/*`, `/api/tenants/:id/effective-capabilities` |
| `TenantApiSingleton` | `tenant` | `tenant` | `/api/tenants/:id/*` (merchant actions) |
| `AdminApiSingleton` | `admin` | `admin` | `/api/admin/*` |
| `AuthenticatedApiSingleton` | `user` | `user` | User profile, preferences |
| `CustomerApiSingleton` | `customer` | `customer` | Customer auth, cart |

**Key insight**: A write in one context (e.g. `tenant:tenant`) never auto-invalidates a cache entry in another context (e.g. `public:public`). You must explicitly cross-invalidate.

## When Cross-Context Invalidation Is Needed

Any time a **merchant/admin write** changes data that a **public endpoint** also serves. Common scenarios:

| Merchant action (write) | Public endpoint (cached read) | Public service to invalidate |
|---|---|---|
| `PUT /api/tenants/:id/bot/config` | `GET /api/public/bot/config?tenantId=:id` | `publicBotService` |
| `POST /api/tenants/:id/bot/avatar` | `GET /api/public/bot/config?tenantId=:id` | `publicBotService` |

## How to Cross-Invalidate

### Pattern 1: Direct public service invalidation (preferred)

Import the public service singleton and call `invalidateCache()` with a wildcard pattern that matches the public cache key. The public service's own `defaultContext`/`defaultIsolation` (both `public`) are used automatically by `invalidateCachePattern`.

```typescript
import { publicBotService } from './PublicBotService';

// After merchant write succeeds:
await publicBotService.invalidateCache(`/api/public/bot/config*${tenantId}*`);
```

The `*` wildcards ensure the pattern matches regardless of the `:public:public` suffix. The `invalidateCache` method in `UniversalSingleton.ts:609-623` detects wildcards and delegates to `invalidateCachePattern`, which uses the service's own `defaultContext`/`defaultIsolation` to target the correct cache partition.

### Pattern 2: When the public service uses a custom cacheKey (not URL)

If the public service passes an explicit `cacheKey` to `makePublicRequest` (e.g. `bot-widget-config-${tenantId}`), the cache key is:

```
bot-widget-config-tid-m8ijkrnk:public:public
```

Invalidate with:
```typescript
await publicBotService.invalidateCache(`bot-widget-config-${tenantId}*`);
```

### Pattern 3: Capability invalidation

`UnifiedCapabilityService` has a dedicated method:
```typescript
import { unifiedCapabilityService } from './UnifiedCapabilityService';
await unifiedCapabilityService.invalidateTenantCapabilities(tenantId);
```

This clears both the in-memory `capCache` and the persistent cache key `unified-caps-${tenantId}`.

## Implementation Checklist

When adding or modifying a merchant write endpoint that affects public data:

1. **Identify the public endpoint(s)** that serve the same data
2. **Find the public service** that fetches that endpoint (grep for the URL pattern in `apps/web/src/services/`)
3. **Determine the cache key** — check if the public service passes an explicit `cacheKey` or relies on the URL
4. **Add cross-invalidation** in the merchant service's write method, after the API call succeeds:
   ```typescript
   await publicService.invalidateCache(`{cache-key-pattern}*${tenantId}*`);
   ```
5. **Also invalidate `UnifiedCapabilityService`** if the change affects feature/capability gates
6. **Test**: Enable verbose logging (uncomment the console.log blocks in `FlexibleApiSingleton.ts`) and verify the public cache key is cleared after the merchant write

## Auto-Invalidation (What It Does and Doesn't Do)

In `FlexibleApiSingleton.ts:1591-1604`:

```typescript
// Auto-invalidate cache for non-GET requests (with login exception)
if (method !== 'GET' && cacheKey) {
  if (url.includes('/api/auth/login')) {
    // skip
  } else {
    if (context || isolation) {
      await this.invalidateCacheWithContext(cacheKey, context, isolation);
    } else {
      await this.clearCache(cacheKey);
    }
  }
}
```

**What it does**: After a PUT/POST/DELETE, invalidates the cache entry for the current request's own cache key in the current context.

**What it doesn't do**: It does NOT invalidate other services' caches, other contexts' caches, or other endpoints that serve the same logical data. That is the responsibility of the calling service method.

## Debugging Cache Issues

### Enable verbose logging

Uncomment the `console.log` blocks in `FlexibleApiSingleton.ts` inside `makePublicRequest` (line ~619) and `makeDefaultRequest` (line ~1046) to see the exact `finalCacheKey` being used.

### Verify invalidation

After a merchant write, check the console for:
1. The PUT request completing successfully
2. The cross-invalidation call executing (look for `[PublicBotService]` or whichever public service's invalidateCache was called)
3. The next GET to the public endpoint should show `Cache MISS` (not `Cache HIT`)

### Common pitfalls

- **Wrong pattern**: If the public service uses a custom `cacheKey` (not the URL), invalidating `/api/public/bot/config*` won't match `bot-widget-config-tid-*`. Match the actual baseKey.
- **Missing tenantId in pattern**: Without `*${tenantId}*` in the pattern, you may invalidate other tenants' caches or miss the target entirely.
- **Circular imports**: If the merchant service and public service would create a circular import, import the public service lazily inside the method body: `const { publicBotService } = await import('./PublicBotService')`.
