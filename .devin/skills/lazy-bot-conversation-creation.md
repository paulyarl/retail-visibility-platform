# Lazy Bot Conversation Creation

## Problem

Every bot widget load created a `bot_conversations` DB record + stored the greeting as a `bot_messages` record — even if the visitor never engaged. This spammed the merchant's conversation list with greeting-only entries. Additionally, a visitor's first message being blocked by guardrails (spam, abuse) would still create a conversation record, polluting the pipeline with junk.

## Solution

**Lazy conversation creation** — a conversation record is only persisted to the database when a visitor sends a message that **passes guardrails**. The flow has three gates before any DB write:

1. **Greeting-only visit** → `prepareConversation` generates a session ID + greeting in memory, returns it to the widget. Zero DB writes.
2. **First message blocked by guardrails** → guardrail check runs before lazy creation. Block response returned without persisting. Zero DB writes.
3. **First message passes guardrails** → conversation record created, user message + bot response stored. One DB write.

## Architecture

### Backend

#### BotConversationService.ts

- **`prepareConversation(params: { tenantId })`** — Generates a session ID via `generateBotConversationSessionId(tenantId)` and fetches the greeting from `bot_configurations`. No DB writes. Returns `{ sessionId, greeting }`.
- **`createConversation(params: { tenantId, sessionId?, ... })`** — Accepts an optional `sessionId`. If provided, uses it; otherwise generates a new one. Creates the `bot_conversations` record + stores the greeting as the first `bot_messages` record.
- **`isSessionValid(sessionId)`** — Returns `true` for sessions without a DB record (pre-creation state). This allows the messages route to accept messages for sessions that haven't been persisted yet.

#### bot-public.ts — POST /conversations

Calls `prepareConversation` instead of `createConversation`. Returns `{ sessionId, greeting }` with zero DB persistence. The `sendMessageSchema` was expanded to accept optional `tenantId`, `embedKey`, `pageContext`, `contextEntityName`, `customerEmail`, `customerPhone` — these are needed for lazy creation on the first message.

#### bot-public.ts — POST /conversations/:sessionId/messages

The message handler flow (in order):

1. **Rate limit check** — `checkRateLimit(sessionId)`
2. **Session validation** — `isSessionValid(sessionId)` (passes for pre-creation sessions)
3. **Resolve tenantId** — from existing conversation, or from request body (`tenantId` / `embedKey`)
4. **Guardrail check** — `guardrailService.checkMessage(tenantId, message)` — runs BEFORE lazy creation
   - If blocked AND conversation exists → persist blocked message (audit trail for active conversations)
   - If blocked AND no conversation → return block response **without persisting** (spam never creates a record)
5. **Lazy conversation creation** — only if message passes guardrails and no conversation exists yet
6. **Store user message** + generate bot response (intent detection, skill execution, RAG, dynamic response)

### Frontend

#### bot-widget.js (vanilla JS embed)

- `sendMessage()` now includes `tenantId`/`embedKey` + `pageContext` in the POST body. These fields are required by the backend to create the conversation on the first message.

#### PublicBotService.ts (React widget service)

- `sendMessage(sessionId, message, context?)` — accepts optional `{ tenantId, pageContext, contextEntityName, customerEmail, customerPhone }` and includes them in the request body.
- Cache keys removed from `startConversation` and `sendMessage` — each call must hit the API fresh (cached session IDs would be stale).

#### PublicBotWidget.tsx (React widget)

- `sendMessage` callback passes `tenantId`, `pageContext`, `contextEntityName`, and pre-chat email/phone to `publicBotService.sendMessage`.

## Key Design Decisions

1. **Guardrails before persistence** — The most important insight. Running guardrail checks before lazy creation ensures spam/abuse never creates a conversation record. This is the difference between "lazy creation" and "lazy creation with abuse prevention."

2. **Audit trail for active conversations** — Once a conversation exists, blocked messages ARE persisted (with `guardrailResult: 'blocked'`). This preserves the audit trail for an ongoing conversation while preventing new spam-only conversations.

3. **Session validity for pre-creation sessions** — `isSessionValid` returns `true` when no DB record exists. This is intentional: the session ID was generated by `prepareConversation` and is valid — it just hasn't been persisted yet. The session expires based on the TTL from when it was created (tracked client-side via the session ID's embedded timestamp or server-side once persisted).

4. **No caching on conversation start** — `startConversation` must always return a fresh session ID. Caching would return a stale session from a previous widget open, which could cause issues if the previous session was already persisted (duplicate session IDs) or expired.

5. **Tenant context in message body** — The frontend must pass `tenantId`/`embedKey` + `pageContext` with every message, not just the first one. The backend uses it only for lazy creation (first message), but including it on every message is simpler and more robust than tracking "first message" state on the client.
