# Quick Start √ó Product Type Awareness Audit

## Executive Summary

The Quick Start product generation system has **zero integration** with the Product Type Awareness system. Every product generated by Quick Start is hardcoded as `physical`, regardless of the tenant's tier capabilities, merchant preferences, or business scenario. This means:

- A tenant on a **digital-only** tier gets physical products they can't properly sell
- A **service business** scenario exists in the UI but has no backend support and still generates physical products
- The AI generation pipeline never asks for or respects product type
- Digital product fields (license_type, download_limit, access_duration_days) are explicitly nulled

## Gap Analysis

### GAP 1: No Product Type Selection in Quick Start UI

**Severity: High**

The Quick Start wizard (`QuickStartWizard.tsx`), modal (`QuickStartProductModal.tsx`), and settings (`QuickStartProductSettings.tsx`) have no UI element for selecting a product type. The user cannot choose whether to generate physical, digital, hybrid, or service products.

**Affected files:**
- `apps/web/src/components/quick-start/QuickStartWizard.tsx` ‚Äî `onGenerate` params have no `productType` field
- `apps/web/src/components/quick-start/QuickStartProductModal.tsx` ‚Äî `QuickStartProductConfig` has no `productType` field
- `apps/web/src/components/quick-start/QuickStartProductSettings.tsx` ‚Äî `ProductGenerationSettings` has no `productType` field

### GAP 2: Backend Hardcodes `physical` Product Type

**Severity: Critical**

The generation engine at `apps/api/src/lib/quick-start.ts:911` hardcodes `productType: 'physical'` in SKU generation and sets all digital product fields to null. There is no parameter to override this.

**Affected code:**
- `apps/api/src/lib/quick-start.ts:909-913` ‚Äî SKU generation hardcoded to `physical`
- `apps/api/src/lib/quick-start.ts:923-926` ‚Äî Digital fields explicitly nulled
- `apps/api/src/lib/quick-start.ts:937` ‚Äî `source: 'MANUAL'` (should reflect AI generation)

### GAP 3: No Capability Check Before Generation

**Severity: High**

The Quick Start route handler (`apps/api/src/routes/quick-start.ts:121-324`) checks:
- ‚úÖ Authentication
- ‚úÖ Tenant permissions
- ‚úÖ `quickstart_enabled` tier feature
- ‚úÖ Rate limits
- ‚úÖ SKU limits
- ‚ùå **Product type capabilities** ‚Äî never calls `ProductTypeService.resolveProductTypeState()`

A tenant on a tier that only allows `digital` products will get `physical` products that violate their tier capability.

### GAP 4: `service_business` Scenario Has No Backend Support

**Severity: Medium**

The frontend scenarios list (`scenarios.ts:169-176`) includes a `service_business` scenario, but:
- It is NOT in the backend Zod validation schema (`quick-start.ts:91-111`)
- It is NOT in the `SCENARIOS` object in `quick-start.ts`
- It is NOT in `BusinessTypeSelector.tsx` BUSINESS_TYPES list
- Submitting it would fail Zod validation with a 400 error

### GAP 5: No Digital Product Data Generation

**Severity: High**

When generating digital products, the system needs:
- `license_type` (personal/commercial/educational/enterprise)
- `access_duration_days`
- `download_limit`
- `external_url` or digital asset
- `delivery_method` (direct_download/external_link/license_key/access_grant)

The Quick Start generation engine has no logic for populating these fields. The AI prompt does not ask for digital product metadata.

### GAP 6: No Service Product Data Generation

**Severity: Medium**

Service products need:
- Duration, availability schedule
- Booking metadata
- No shipping required
- Potentially unlimited stock (handled in ProductTypeStep but not in Quick Start)

### GAP 7: No Hybrid Product Data Generation

**Severity: Medium**

Hybrid products need both physical inventory fields AND digital product fields populated.

### GAP 8: SKU Generator Always Gets `physical`

**Severity: Medium**

The `generateAutoSKU()` call at `quick-start.ts:909` passes `productType: 'physical'`, so generated SKUs always have physical-type prefixes. Digital products should get `DIGI` prefixes, services should get `SERV`, etc.

### GAP 9: No Product Type in AI Generation Prompt

**Severity: Medium**

The `ProductCacheService.getProductsForScenario()` call at `quick-start.ts:773-780` passes `businessType` and `categoryName` but not `productType`. The AI prompt for product generation doesn't know whether to generate digital product metadata, service descriptions, etc.

### GAP 10: No Product Type Validation on Created Items

**Severity: Low**

After generation, items are created directly in the database without checking whether the product type is allowed for the tenant. The regular item creation API (`items.ts` route) goes through capability checks, but Quick Start bypasses this by calling `prismaClient.inventory_items.create()` directly.

## Recommended Fix Plan

### Phase 1: Backend Product Type Awareness (Minimal)

**Goal:** Quick Start respects tenant's product type capabilities.

1. **Add `productType` to `QuickStartOptions` interface** in `quick-start.ts`
2. **Add `productType` to Zod schema** in `quick-start.ts` route (default: `'physical'`)
3. **Call `ProductTypeService.resolveProductTypeState()`** in the route handler before generation
4. **Validate requested product type** against tier `allowedTypes`
5. **Pass `productType` to `generateQuickStartProducts()`** and use it for:
   - SKU generation (`productType` parameter)
   - Stock defaults (digital/service ‚Üí 9999)
   - Digital product fields (populate for digital/hybrid)
6. **Add `service_business` to backend SCENARIOS** and Zod schema

### Phase 2: Frontend Product Type Selection

**Goal:** User can choose product type in Quick Start UI.

1. **Add product type selector** to `QuickStartProductModal` and `QuickStartProductSettings`
2. **Use `useProductTypeCapability` hook** to show only allowed types
3. **Add `productType` to `QuickStartProductConfig` and `ProductGenerationSettings`**
4. **Default to tenant's effective product type** (from capability resolution)
5. **Show upgrade messaging** for types not in tenant's tier

### Phase 3: AI Generation Per Product Type

**Goal:** AI generates appropriate metadata for each product type.

1. **Pass `productType` to `ProductCacheService.getProductsForScenario()`**
2. **Update AI prompt** to generate type-specific metadata:
   - Digital: license type, download limit, access duration, delivery method
   - Service: duration, booking notes, no shipping
   - Hybrid: both physical and digital fields
3. **Populate digital product fields** in `generateQuickStartProducts()` based on type
4. **Set `source: 'AI_QUICK_START'`** instead of `'MANUAL'`

### Phase 4: Service Business Scenario

**Goal:** `service_business` scenario generates service-type products.

1. **Add `service_business` to SCENARIOS** in `quick-start.ts` with service categories
2. **Add to Zod schema** in route
3. **Add to `BusinessTypeSelector.tsx`** BUSINESS_TYPES
4. **Force `productType: 'service'`** when scenario is `service_business`
5. **Generate service-appropriate products** (consultations, repairs, appointments)

### Phase 5: Validation & Constraints

**Goal:** Generated products pass all capability checks.

1. **Run CCL constraint validation** on generated products
2. **Validate each product** against tenant's effective capabilities
3. **Add product type to Quick Start eligibility check** endpoint
4. **Return product type info** in generation result

## Key Design Decisions

1. **Default product type**: Use tenant's `effective_type` from capability resolution. If `flexible`, default to `physical`. If single type (e.g. `digital`), default to that type.

2. **Service business scenario**: Should auto-select `service` product type and hide the type selector (or lock it to service).

3. **Digital product generation**: AI should generate plausible digital products (e-books, software licenses, digital art, templates) with appropriate metadata. For the `books_media` scenario, digital products could be e-books. For `electronics`, could be software/firmware.

4. **Hybrid products**: Generate physical products that include a digital component (e.g. a book with a digital download code, hardware with companion app).

5. **Backward compatibility**: Existing Quick Start behavior (physical-only) should remain the default when no product type is specified. This ensures no breaking change for existing tenants.

---

## Conclusion

The Quick Start system was built before the Product Type Awareness system and was never updated to integrate with it. The product creation wizard (`ProductTypeStep.tsx`) is the reference implementation for how product type selection should work with capability gating. The fix is straightforward: wire the same capability checks into Quick Start, add a type selector to the UI, and update the generation engine to produce type-appropriate data.

The `service_business` scenario in `scenarios.ts` is a forward-looking UI addition that was never completed on the backend. It should be either fully implemented or removed to avoid confusion.

## Propagation Gap Analysis

### PROP GAP 1: Social Commerce Feeds Don't Propagate Product Type

**Severity: High**

All three feed generators (Google, Meta, TikTok) query `inventory_items` but **do not select `product_type`**. This means:

- Digital products get synced to Google Shopping / Meta Commerce / TikTok Shop as if they were physical products
- Service products get synced with shipping dimensions and stock quantities that don't apply
- No filtering ‚Äî digital-only or service-only tenants have all their products synced to channels that may not support those product types

**Affected files:**

- `apps/api/src/lib/google/feed-generator.ts:30-55` ‚Äî `select` clause omits `product_type`; `FeedItem` interface has `productType` field but it's set to `undefined` at line 61
- `apps/api/src/lib/meta/feed-generator.ts:42-63` ‚Äî `select` clause omits `product_type`; `MetaFeedItem` interface has no product_type field at all
- `apps/api/src/lib/tiktok/feed-generator.ts:36-57` ‚Äî `select` clause omits `product_type`; `TikTokFeedItem` interface has no product_type field at all

### PROP GAP 2: Digital Product Fields Missing from Feed Generators

**Severity: High**

Digital products require different feed attributes:
- **Google Shopping**: `is_bundle`, `identifier_exists` = false, no `shipping` required
- **Meta Commerce**: `availability` semantics differ (digital = available vs. physical stock-based)
- **TikTok Shop**: Digital products need different `package_weight`/dimensions (zero or N/A), different category mapping

None of the feed generators select or propagate digital-specific fields:
- `digital_delivery_method`
- `license_type`
- `access_duration_days`
- `download_limit`
- `digital_assets`

### PROP GAP 3: Service Products Inappropriately Synced to Commerce Channels

**Severity: Medium**

Service products (appointments, consultations) should not be synced to Google Shopping, Meta Commerce, or TikTok Shop as physical products. They:
- Have no shipping dimensions
- Have no physical stock
- Use booking/scheduling rather than checkout
- May need different availability semantics

**Current behavior**: All three feed generators include any `item_status='active' AND visibility='public'` item regardless of `product_type`.

### PROP GAP 4: Materialized View Digital Field Propagation

**Severity: Medium**

The materialized view (queried by `BotProductCatalogService`, `recommendationService`, `FeaturedService`, `ScopeRouter`) includes `product_type` but may not propagate digital-specific fields (`license_type`, `digital_delivery_method`, `digital_assets`, `access_duration_days`, `download_limit`). This means:

- Bot product search results don't show digital product delivery details
- Recommendations don't factor in digital product characteristics
- Storefront display may not show digital product metadata

### PROP GAP 5: Variant Creation Doesn't Inherit Digital Fields

**Severity: Medium**

When variants are created via `VariantService` or `VariantBulkOperationsService`, the parent's `product_type` is used for SKU generation but **digital product fields are not inherited**:
- `license_type`
- `access_duration_days`
- `download_limit`
- `digital_delivery_method`
- `digital_assets`

A digital product with variants would have child variants missing digital fulfillment data.

### PROP GAP 6: Quick Start Sets `product_type` to NULL (Root Cause)

**Severity: Critical**

The Quick Start generation engine at `apps/api/src/lib/quick-start.ts:906-947` creates `inventory_items` records without setting `product_type` on the item data object. The field is absent from `itemData`, so it defaults to `NULL` in the database.

Every downstream consumer uses `|| 'physical'` fallback logic:
- `SingleProductService.ts:391` ‚Äî `product.product_type || 'physical'`
- `StockService.ts:32` ‚Äî `item.product_type || item.inventory_items?.product_type`
- `checkout.ts:342` ‚Äî `inventoryItem.product_type || 'physical'`
- `recommendationService.ts:656` ‚Äî `row.product_type || 'physical'`
- `FeaturedProductsSingletonService.ts:490` ‚Äî `mvProduct.product_type || 'physical'`
- `TierBasedLandingPage.tsx` ‚Äî `product.productType` (undefined if not propagated)

This means Quick Start products silently become "physical" everywhere, even if the tenant's tier only allows digital.

### PROP GAP 7: Bot Product Catalog Doesn't Filter by Product Type

**Severity: Low**

`BotProductCatalogService.ts` queries `mv_storefront_discovery` and returns `product_type` in results, but doesn't filter by type. A customer asking a bot about "digital downloads" gets all products regardless of type. The bot's `formatProductContext` doesn't include digital-specific metadata in the context injected to GPT.

### PROP GAP 8: Recommendation Service Ignores Product Type Compatibility

**Severity: Low**

`recommendationService.ts` returns `product_type` in results but doesn't use it for compatibility scoring. A customer viewing a digital product might get recommended physical products that require shipping, or vice versa.

## Recommended Propagation Fix Plan

### Phase 6: Quick Start Sets `product_type` on Creation (Root Cause Fix)

**Goal:** Every Quick Start product has a non-null `product_type`.

1. **Add `product_type` to `itemData`** in `apps/api/src/lib/quick-start.ts:906-947`

   - `service`: Set stock to 9999, no shipping fields
   - `hybrid`: Set both physical and digital fields
   - `physical`: Null out digital fields (current behavior)

### Phase 7: Social Commerce Feed Product Type Propagation

**Goal:** Feeds correctly propagate product type and type-specific fields.

1. **Google feed-generator.ts**:

## Propagation Design Decisions

1. **Service products in feeds**: Filter OUT service products from Google/Meta/TikTok feeds. These channels don't support service-type listings. Service products should be storefront-only.

2. **Digital products in feeds**: Include in feeds but with type-appropriate attributes (no shipping, different availability semantics, `identifier_exists` = false for Google).

3. **Variant digital inheritance**: Variants should inherit digital fields from parent at creation time. If a merchant later changes the parent's digital settings, they must manually update variants (or we add a "sync to variants" action).

4. **MV digital fields**: Adding digital fields to the MV is the highest-leverage fix ‚Äî it automatically propagates to bot, recommendations, featured, and storefront API.

5. **Null vs. 'physical'**: The root cause is Quick Start creating items with `product_type = NULL`. Every downstream consumer falls back to `'physical'`. Fixing Quick Start to explicitly set `product_type` eliminates the null-fallback pattern.

6. **Feed filtering strategy**: Add a `shouldSyncToChannel(productType, channel)` utility function that each feed generator calls. Centralizes the filtering logic.

## Propagation Gap Analysis

### ORG GAP 1: Sync-from-Hero Missing Product Type Fields (CRITICAL)

**Severity: Critical**

The `POST /:id/sync-from-hero` route at `apps/api/src/routes/organizations.ts:2051-2099` creates new items but **omits all product type and digital fields**:

**Missing from sync-from-hero create:**
- `product_type`
- `digital_delivery_method`
- `digital_assets`
- `access_duration_days`
- `download_limit`
- `license_type`
- `sale_price_cents`
- `payment_gateway_type`
- `payment_gateway_id`
- `features`
- `specifications`
- `enhanced_description`
- `has_variants`
- `is_featured`
- `featured_at`
- `featured_until`
- `featured_priority`
- `featured_type`

This means:
- A digital product propagated from hero gets `product_type = NULL` ‚Üí silently becomes "physical" via `|| 'physical'` fallbacks
- All digital metadata (license type, download limit, access duration, delivery method, digital assets) is lost
- Enriched content (features, specifications, enhanced_description) is lost
- Featured/badge data is lost
- Sale pricing is lost
- Payment gateway configuration is lost
- Variant parent flag (`has_variants`) is lost ‚Äî variants won't be propagated

**Contrast:** The single propagate and bulk propagate routes DO copy these fields correctly (lines 883-888 for update, 1109-1114 for create, 1558-1563 for bulk update, 1772-1777 for bulk create).

### ORG GAP 2: No Product Type Capability Check on Target Tenants

**Severity: High**

None of the three propagation routes check whether the target tenant's tier allows the product type being propagated. A digital product can be propagated to a tenant on a tier that only allows physical products.

**Affected routes:**
- `POST /:id/items/propagate` ‚Äî no `ProductTypeService.resolveProductTypeState()` call for target tenants
- `POST /:id/items/propagate-bulk` ‚Äî same
- `POST /:id/sync-from-hero` ‚Äî same

**Example failure case:**
1. Organization has 3 locations: Location A (professional tier, allows digital), Location B (starter tier, physical only), Location C (storefront tier, physical + digital)
2. Propagate a digital product from A ‚Üí B and C
3. Location B receives a digital product it can't properly sell or fulfill
4. The product appears in B's storefront but digital fulfillment will fail

### ORG GAP 3: Variant Propagation Doesn't Copy Digital Fields

**Severity: Medium**

The `propagateVariants()` function at `apps/api/src/utils/variant-propagation.ts:138-157` creates variant records but doesn't copy digital product fields to variants:

Note: This gap depends on whether `product_variants` table has these columns. If variants inherit digital behavior from parent, this may be by design. But if variants can have their own digital configuration, this is a gap.

### ORG GAP 4: No Product Type Awareness in Propagation UI

**Severity: Low**

The propagation modals (`PropagationModal.tsx`, `BulkPropagationModal.tsx`, `PropagateItemModal.tsx`) don't display or filter by product type:

**Severity: Low**

The `ensureGlobalCatalogEntry()` call during propagation (lines 984-1002, 1638-1656) doesn't pass `product_type` or digital fields. The global catalog entry will lack product type information, which could affect availability lookups and cross-tenant catalog features.

### ORG GAP 6: Sync-from-Hero Missing Variant Propagation

**Severity: Medium**

The sync-from-hero route at lines 2029-2135 creates items but **never calls `propagateVariants()`**. If a hero location product has variants, those variants are not propagated to target tenants. ## Organizational Propagation Design Decisions

1. **Sync-from-hero should mirror single propagate:** The sync-from-hero route should copy the exact same field set as the single propagate create mode. This is a bug fix, not a feature addition.

2. **Capability check is a warning, not a hard block:** For backward compatibility, propagate product type mismatches as "skipped" with a clear reason, not as errors. Tenants can then upgrade their tier or contact support.

3. **Bulk create enriched content fix is independent:** The missing `features`, `specifications`, `enhanced_description` in bulk create mode is a bug that should be fixed regardless of product type awareness.

4. **Variant digital fields depend on schema:** Before implementing Phase 15, verify whether `product_variants` table has columns for digital fields. If not, variants inherit from parent by design and no change is needed.

5. **Sync-from-hero should support update mode:** Currently it only creates (skips if SKU exists). Adding update mode would make it consistent with single/bulk propagate. This is a separate enhancement but should be considered alongside the field fix.
